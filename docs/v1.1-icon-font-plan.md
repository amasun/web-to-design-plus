# Icon Font 转 SVG 抓取 - 技术可行性与实施计划

## 技术可行性分析 (Feasibility)
**结论：完全可行，但具有一定技术复杂度。**

在浏览器原生 API 中，**无法**直接将一个加载的字体字符提取为矢量路径（SVG Path）。浏览器只能将文字渲染为光栅图像（Canvas PNG）或直接作为文字节点渲染。要将 Icon Font 真正转化为真实的矢量 SVG，我们必须**在前端解析字体文件（TTF/WOFF）并手动提取对应的字形（Glyph）轮廓数据**。

为了实现这一点，我们需要引入一个能够解析字体文件的第三方 JavaScript 库，最成熟的选择是 **[OpenType.js](https://opentype.js.org/)**。

---

## 核心实现步骤 (Implementation Steps)

### Phase 1: 智能识别 Icon Font (DOM & CSS 嗅探)
在现有的 `svg-grabber.js` 扫描流程中，加入对 Icon Font 的专属嗅探逻辑：
1. **特征匹配**：遍历 DOM 树，寻找带有常见图标类名（如 `fa-*`, `iconfont`, `icon-*`, `material-icons`）的元素。
2. **伪元素探测**：通过 `window.getComputedStyle(el, '::before')` 获取元素的 `content` 属性。Icon Font 通常通过 `content: "\e601"` 注入 Unicode 字符。
3. **记录元数据**：提取出该字符的 **Unicode 值**，以及对应的 **Font-Family 名称** 和 **元素类名**（用作导出的文件名）。

### Phase 2: 字体文件寻址与加载 (Font Fetching)
1. 遍历 `document.styleSheets` 和 `CSSRule.FONT_FACE_RULE`，匹配在 Phase 1 中获取到的 Font-Family。
2. 提取 `@font-face` 中的 `src` URL（优先选择 WOFF / TTF 格式）。
3. 使用 `fetch(url).then(res => res.arrayBuffer())` 将真实的字体文件下载到内存中。
   *(注：此处可能需要利用我们现有的跨域代理机制或 background.js 来绕过 CORS 限制)*

### Phase 3: 字形到矢量的数学转换 (Glyph to Vector via OpenType.js)
1. 在 `build.js` 和插件清单中引入压缩版的 `opentype.js`。
2. 使用 `opentype.parse(buffer)` 解析字体文件。
3. 调用 `font.charToGlyph(unicodeChar)` 获取对应的字形。
4. 调用 `glyph.getPath(x, y, fontSize)` 生成标准 SVG `<path>` 的 `d` 属性指令。

### Phase 4: 注入 SVG Grabber 面板 (Integration)
1. 将解析出的 SVG Path 包装成标准的 `<svg viewBox="..."> <path d="..."/></svg>` 标签。
2. 将颜色（`color` CSS 属性）填充至 SVG 中。
3. 将该 SVG 资产无缝插入现有的 `svg-grabber` 结果池中。
4. 在 UI 呈现上，它们会像普通的图片或内联 SVG 一样展示，支持复制源码或点击下载。

## Open Questions (需要您决策的问题)

> [!WARNING]
> **关于引入第三方库的包体积问题**
> 
> `opentype.js` 是一个非常强大的库，但它的压缩版大小约为 **150KB - 200KB**。虽然对于 Chrome 插件来说这个体积完全可以接受，但这违背了我们此前“尽可能零依赖、轻量级 Vanilla JS”的纯粹架构初衷。
> 
> **您是否同意在此功能中引入 `opentype.js` 作为依赖？**
> - **选项 A**：同意引入。Icon 转 SVG 需求很痛点，体积增加 200KB 是值得的。（如果选 A，我们需要把 `opentype.js` 下载到 `src/lib/` 目录下并参与打包）。
> - **选项 B**：不同意引入。保持极简架构，放弃真实矢量提取，只做纯文字级别的嗅探（或者暂不开发此功能）。
